# 数据导入测试与开发协同规范

本文档定义“导入数据测试流程”和“开发流程”的协同方式，目标是：  
在持续开发期间，稳定推进真实数据导入，不因功能迭代导致导入质量回退。

## 1. 目标与原则
- 先可验证，再写入：所有导入必须先 `preview`，再 `dry_run`，最后正式执行。
- 小批次推进：先样本、再子集、再全量，避免一次性导入造成大面积错误。
- 可回溯：每次导入都必须能定位到 `import_batch_id` 和源文件。
- 可回滚：正式导入前必须先备份数据库。
- 开发与测试交替：每完成一段导入逻辑改动，必须马上跑一轮导入测试。

## 2. 协同节奏（推荐）
采用简单的“开发一小步 -> 导入测试一小步”节奏：

1. 需求确认（半天内）
- 明确本轮要支持的 Excel 模板/字段差异。
- 定义通过标准（哪些字段必须成功，哪些错误可接受）。

2. 开发实现（当天）
- 修改解析/映射/校验逻辑。
- 本地运行 `scripts/smoke_test.py`，确保核心流程不回归。

3. 导入测试（当天）
- 在测试库执行 `preview` + `dry_run`。
- 核对错误报告与预期是否一致。
- 修复问题后重复步骤 2-3，直到通过。

4. Windows 验证（次日或同日）
- 打包后在 Windows 环境按真实操作路径再跑一次测试。
- 通过后才允许进入真实导入窗口。

## 3. 三阶段导入门禁（Gated Import）

### Gate A: 预览（Preview）
接口：`POST /import/inbound/preview`

通过标准：
- 能识别表头行（`header_row` 有值）
- 字段映射合理（`field_mapping` 覆盖关键字段）
- 抽样行内容与原文件一致

### Gate B: 预演（Dry-run）
接口：`POST /import/inbound/execute`，`dry_run=true`

通过标准：
- 接口执行成功
- `failed_rows` 在可接受范围内（初期建议 <= 5%，后续收紧到 <= 1%）
- 错误原因可解释（客户未映射、缺必填字段等）

### Gate C: 正式导入（Commit）
接口：`POST /import/inbound/execute`，`dry_run=false`

前置条件：
- 已完成数据库备份（`POST /backup` 或 `python scripts/backup_db.py`）
- Gate A/B 均通过
- 业务方确认本批次可导入

导入后动作：
- 记录 `batch_id`
- 执行对账检查（见第 6 节）

## 4. 测试数据分层（简单且实用）
- `L0 冒烟样本`：10-30 行，覆盖最小闭环（已有 `scripts/smoke_test.py`）。
- `L1 常规样本`：100-500 行，模拟日常数据。
- `L2 脏数据样本`：包含空客户名、客户别名未映射、数值异常、续行等问题。
- `L3 真实子集`：从真实数据抽 1 天或 1 客户子集，作为上线前验证。

要求：
- 每次修改导入逻辑，至少跑 `L0 + L2`。
- 每次准备正式导入，必须跑 `L3`。

## 5. 开发与测试同步规则
- 任一导入相关代码变更（`app/services/importer.py`、映射逻辑、客户别名规则）后，必须同步更新测试样本或测试说明。
- 发现生产样本新格式时，先补测试样本，再改代码。
- 不允许“只改代码不测导入”或“只测不固化规则”。

## 6. 每批导入后的最小核对清单
导入完成后至少做以下核对：

1. 批次统计核对
- 查询 `import_batches`：`total_rows/success_rows/failed_rows` 是否符合预期。

2. 在库数据核对
- 抽查导入记录的 `customer_id`、`item_name_cn`、`cbm_calculated/cbm_override`。
- 确认状态默认为 `IN_STOCK`（如业务未分配出柜）。

3. 客户映射核对
- 对失败行中的客户名，确认是否需要补充 `customer_aliases`。
- 补充别名后重新 `dry_run` 验证。

4. 业务可见性核对
- 在 Web 页“查询”模块检查是否可检索到导入数据。

## 7. Windows 实操流程（建议）
1. 运行 `start_windows.bat` 启动系统。  
2. 先执行一次备份。  
3. 在“导入导出”模块执行 `preview`。  
4. 执行 `dry_run=true`。  
5. 核对结果并修复映射（必要时维护别名）。  
6. 执行 `dry_run=false` 正式导入。  
7. 到“查询”模块核对导入结果。  

## 8. 异常处理策略
- 失败率超阈值：停止正式导入，仅保留 `dry_run` 结果，先修映射/规则。
- 导入后发现严重错误：立即停止后续业务操作，使用最近备份恢复，再修复后重导。
- 同一问题重复出现 2 次以上：将该规则固化到解析逻辑或别名字典，避免人工重复处理。

## 9. 验收标准（导入能力）
- 能稳定完成 `preview -> dry_run -> commit` 全链路。
- 常规样本（L1）成功率 >= 99%（已知脏数据除外）。
- 导入后查询与导出链路可用。
- 每次导入都有记录、可追踪、可复盘。

## 10. 下一步执行建议
1. 先建立一批 `L2` 脏数据样本（10-20 个典型错误）。
2. 先在 Windows 跑 3 轮 `preview + dry_run`，再做第一轮正式小批量导入。
3. 每轮导入后把问题归类为“映射缺失/模板差异/数据质量”，按类别修复。
